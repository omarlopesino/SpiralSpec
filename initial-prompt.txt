  ## Tasks
       - tasks/ : Folder with all the tasks needed to implement the task. This folder must be created and filled by the agent. The agent reads the three main files to figure out what tasks are needed.

  ### Frontmatter properties

  - *name*: Title of the task.
  - *ground*: Each task can be base or incremental. Base means that it can be done without  done before. Incremental means it depends from one several tasks being donebefore. So, ground property must have null value if it is base, and the name of the task, which must be equal to a task of this spec as a reference
  - *goal*: Each task must complete a specific objective. It must be enough large to don't e phrase, and enough short to don't write the full spec in one file. As thisproject is for developers, feel free to define every technical action.

  - *status*: These are the possible statuses:
     - backlog: The task has been defined by the AI, but not approved by the developer.
     - todo: The task has been approved by the user and ready to start
     - inprogress: The task is being implemented by the AI.
     - verification: It needs to be tested by the user.
     - release:
     - done:
  ### File contents

    - # Context with the minimal needed context for the task.

  # Phases of an spec

  Phases are the phases an spec can have:

  - *Definition*: the user must define the context, acceptance criteria and solution. This must be done by the user. Probably the user will copy and paste from an issue, or use an auxiliar skill to
  write it. There must exist help to create the folder with the empty files, with a command
  - *Plan*: The agent must read all the user provided files and create the task.md files at the end. Agent must find gaps or caveats and ask user about them. The developer msut guide the agent to solve
  the doubts, to assume something if it is needed, or to postpone it. With this, the agent can be done now, and wait for the rest of tasks. As the artifacts contain enoughinformation to know the task status and the next steps, this allows the agent to be autonomous. The agent must start with base tasks and continue with incremental tasks. After a task is defined the status file next steps must be edited . After updating status, the developer must decide to straight implement the task, to define more tasks if needed (in this case the agent msut suggest before what tasks can be done), or refine the task.

  All tasks must be created with backlog status.

  - *Plan review*
  In this phase , user can provide feedback to the created tasks planned by the AI. user must refine and refine with feedback until the plan is as solid as it needs in this specific moment.

  - *Implementation*: This is a stage where the coding agent implement the task. Agent starts when the user decides. It is fully autonomous. If it find gaps along the way it asks the user.

  If user wants AI to be autonomous , it tries to make a solution by itself but reports in tion imaginated by the agent. This is only valid when the solution is completelylogic or it is not complex. If it is a complex arquitecture change, or it has too many caveats, or it is prone to be discarded, task must be stopped. In the status file, for the specific task, it must be specified that the task could not be done autonomously and why, along with proposed solutions. In the next steps just add a brief line explaining that the user must help to unblock the task.

  AI can only implement tasks that are in todo / inprogress status. The rest of states require user action.
  When a task is started to be implemented , it must be changed from todo to inprogress.
  When a task is ready to verify, the status of the task must change from inprogress to ver
  Before movig to verification, the self agent must do the verification itself to don't fall in speculations.

  - *Verification*: This is the manual step where the user verify tasks. user will read status file to know what tasks needs to review, then he will lookup for the task and read the testing details. after veryfying he can tell the agent the task is done and the agent will change the task status to release and update the status readme and deployment files.

  If there are errors or iterations needed, the user can provide feeedback. The agent will the status readme file if needed. It can also update the task if needed, or theincremental tsaks that are related to this task. It can also update ground tasks. It can also argue the acceptance criteria is not complete and update it. It can update the main context if more
  context is needed.
